HyperBeetle

HyperBeetle

HyperBeetle is a fan-made tribute to Thumper. It supports custom levels and modding.

License

HyperBeetle source and documentation are released under the GNU General Public License v3.0.

Building

This project has all dependencies either directly in the source tree or as git submodules. Make sure you clone with submodules. --recurse-submodules.

Uses Premake. Have this installed.

premake5 vs2022 to generate a Visual Studio solution for Windows.

premake5 gmake2 to generate makefiles.

Custom GLSL syntax

HyperBeetle uses a modified version of glsl. This is to allow shaders to be reasoned about much easier.

Motivation

  • Reduce the requirement to have separate shaders for each shader stage.
  • Allow us to reuse code and use features such as #include.

Features

Function annotations

Functions may use C++ style annotations. [[vert]] and [[frag]]. Annotation a function with one of these will only include it in that shader type.

[[vert]] void main(void) {}
[[frag]] void main(void) {}

Tokens

Tokens match premake syntax for tokens. They are simply placeholders that get swapped at compiletime.

  • %{engine_version} MAY get changed to 1.0.0.

in / out / varying modifications

Varying was removed when other shader types such as geometry shaders were introduced. HyperBeetle only supports vertex and fragment shaders.

in and out type qualifiers are conditionally removed based on shader stages. This changes their semantic meaning.

  • in now ALWAYS refers to a vertex shader input.
  • out now ALWAYS refers to a fragment shader output.

Varying is used to transfer values between the two shaders. These will be changed to the correct type qualifier at runtime.

in vec3 inPosition = 0; // inout assignment
in vec2 inTexCoord = 1; // inout assignment

varying vec2 vTexCoord;

out vec4 outColor = 0; // inout assignment

uniform sampler2D uAlbedo;

[[vert]] void main(void) {
    gl_Position = vec4(inPosition, 1.0);
    vTexCoord = inTexCoord;
}
[[frag]] void main(void) {
    outColor = texture(uAlbedo, vTexCoord);
}

in / out assignment

Layout qualifiers are not supported. Instead we assign an int to the input and output types. This will generate the associated layout qualifers, or use a fallback.

Extra preprocessor defined

Since the shader stages are combined. HB_VERT is defined if the source is being used as vertex shader code. and HB_FRAG is defined if the source is being used as fragment shader code. HB_GENERATED will be defined if the code was generated by the engine preprocessor.

Removed version directive

The version may not be specified. The engine will do this for you.

Include support

You may include other code by using #include <name>. This does NOT refer to a file name. This refers to the name assigned to the a source code block in the engine.

ARB_shading_language_include is not used to implment this feature.

Profiling

HyperBeetle uses the Tracy Profiler. A Profiler is used to gather detailed frame information. This helps us to find hot spots in our code.

To use the profiler simply attach Tracy to HyperBeetle.

RenderDoc

We have built in support for renderdoc. Meaning if the application is luanched via renderdoc, we can interact with renderdoc. Additionally if the application is launched outside of renderdoc. The engine will attempt to automcatically attach it.

Renderdoc search other

This applies when the application was not ran via renderdoc. Windows currently supports this better.

Windows

  • renderdoc.dll
  • %ProgramFiles%/RenderDoc/renderdoc.dll

Unix

  • librenderdoc.so

Ui changes

The application disables the renderdoc gui. This ui gets quite intrusive when using the builtin tooling. Renderdoc has a debug ui which may be accessed in the engine.

Frame capture

F12 and PRTSC are commonly used to take screenshots, thus these defaults of RenderDoc are changed. Use F2 to take a frame capture.

Opening captures

The debug ui will contains a list of captures. If you're running windows. You can simply click an item in this list to open renderdoc.

Fonts

Thumper uses the Futura_PT font. Due to liscencing this font cannot be redistributed.

The first font loaded should be the english font. The first loaded font is conditionally chosen.

First font load

  1. fonts/Futura_PT_Heavy.ttf
  2. fonts/NotoSans-Bold.ttf

Additionally more fonts may be loaded to support more languages. Currently fonts/NotoSansKR-Bold.ttf is loaded.

To do something else edit the Fonts global inside scripts/config.lua.
